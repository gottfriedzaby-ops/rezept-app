## Summary

The app is substantially more mature than its top-level docs claimed: i18n
(de/en/nl via next-intl), a Serwist PWA with offline fallback, PDF import,
account-to-account library sharing, an admin dashboard, and push-notification
plumbing are all shipped. Code quality is high — strict TypeScript, 48 green
test suites (547 tests), consistent `{ data, error }` API contract, RLS on all
user-facing tables, and carefully engineered Claude prompts.

The audit found **two unauthenticated API endpoints**, a family of
**N+1 / pagination bugs** that break at >50 registered users, **no CI
pipeline**, and a set of UX gaps (no loading skeletons, no visible focus
indicators, hardcoded strings bypassing i18n, no toasts, no dark mode).
Scalability of the recipe list (fetch-everything + client-side filtering) is
the main architectural limit for a public launch.

### Corrections to earlier assumptions (verified)

- `PATCH /api/settings` already uses `upsert … onConflict: "user_id"`; only GET races.
- The Jina Reader and og:image fallback fetches in `import-url` already have
  timeouts; only the primary `fetchOnce()` lacked one.
- `RecipeCover.tsx` correctly uses `next/image` for photos and plain `<img>`
  only for local decorative SVGs — no change needed there.
- `messages/de.json` already contained some keys (e.g. `Common.allRecipes`)
  that pages hardcoded instead of using.

## Notable strengths (keep doing this)

- **RLS everywhere it matters** — all user-facing tables ship policies in the
  same migration that creates them.
- **Claude prompt engineering** (`lib/claude.ts`) — metric-priority rules,
  per-portion division, JSON repair, review pass with section-order safeguard.
- **Test discipline** — 48 suites / 547 tests with documented mock patterns
  (`docs/test-concept.md` §15); hermetic Playwright setup.
- **URL-as-state** filtering UX in `RecipeList.tsx` (shareable, back-button
  friendly) — worth preserving when search moves server-side.
- **Lazy-proxy Supabase clients** (`lib/supabase.ts`) keep `next build`
  working without env vars.
- **Mobile-first polish** — safe-area insets, wake lock in cook mode,
  full-screen shopping mode with haptics.

## Verification constraints of this audit branch

This branch was developed in an environment without database credentials or
Docker, so **SQL migrations are authored but not applied**. All code depending
on the two new tables (`profiles`, `meal_plan_entries`) degrades gracefully
(Postgres `42P01` → legacy fallback / empty state) until the operator applies
the migrations — see the checklist in [roadmap.md](./roadmap.md). Live RLS,
the `auth.users` sync trigger, real uploads, and Claude calls could not be
exercised here; unit/integration suites, lint, and a production build all run
green.
